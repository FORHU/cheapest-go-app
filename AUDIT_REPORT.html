Phase 1 — Booking Reliability

Hotel & flight booking confirmation flows, payment handling, state machines

CRITICAL   success: true returned when DB save fails

src/lib/server/bookings.ts:268–302

confirmAndSaveBooking() calls LiteAPI to confirm the hotel booking, then attempts an atomic DB save via RPC. If the DB save fails (either rpcError or a thrown exception), the function returned success: true with an error message string. The API route treated this as a successful booking — never triggering a Stripe refund, creating a success notification, and returning a success response to the client.

Impact: The user sees "Booking Confirmed" but the booking never appears in /trips because it's not in the database. The hotel IS booked and payment IS charged, but there's no record in the system.

// BEFORE (bug):
return {
  success: true,  // ← WRONG: DB failed, but tells caller it succeeded
  data: { bookingId, status, ... },
  error: 'Booking confirmed but failed to save...',
};

FIXED Changed to success: false with liteApiConfirmed: true flag. Route now handles this case separately — does NOT refund Stripe (hotel is booked), creates CRITICAL admin notification for manual reconciliation.

CRITICAL   No double-click prevention on hotel booking confirm

src/lib/server/bookings.ts:150–180

No concurrency guard existed on confirmAndSaveBooking(). If the user double-clicks the "Confirm" button, two parallel requests reach LiteAPI, potentially creating two hotel bookings for the same room. Unlike flights (which have idempotency keys and a state machine), hotels had zero protection.

Impact: Duplicate hotel bookings charged to the user's card. LiteAPI doesn't deduplicate by prebookId.

FIXED Added in-memory Set<string> guard keyed by prebookId. Second concurrent request for same prebookId is immediately rejected. Set is cleaned in finally block.

HIGH   Route refunds Stripe even when hotel IS booked (after DB failure)

src/app/api/booking/confirm/route.ts:55–68

After fixing the success: true bug above, confirmAndSaveBooking now returns success: false for DB failures. But the route's existing logic would then fall through to the Stripe refund block — refunding the user even though the hotel booking IS confirmed in LiteAPI. This would mean: hotel booked, user refunded, company eats the cost.

FIXED Added result.liteApiConfirmed check before the Stripe refund block. DB-failure-after-LiteAPI returns HTTP 500 with booking data + error message. Stripe is NOT refunded. Admin is notified via createNotification with CRITICAL severity.

MEDIUM   No DB-level idempotency for hotel bookings

src/lib/server/bookings.ts

The bookings table does not store the prebookId, so there's no way to check if a booking for a given prebook was already created. Full idempotency would require adding a prebook_id column with a unique constraint — but this was out of scope (no DB migrations).

Status: Documented. The in-memory concurrency guard mitigates the primary risk (double-click). Server-restart race conditions remain a theoretical edge case.

MEDIUM   No state machine for hotel bookings

Unlike flight bookings (which have a state machine: pending → booked → awaiting_ticket → ticketed), hotel bookings have no formal state transitions. The status field is set ad-hoc ('confirmed', 'cancelled', 'cancelled_refunded') without validation of allowed transitions.

Status: Documented. Implementing a state machine would require application code + potentially DB constraints. Low practical impact since hotel lifecycle is simpler than flights.

Phase 2 — Post-Booking & Email Reliability

Email notifications, webhook handlers, post-booking automation

CRITICAL   Hotel booking: NO confirmation email sent to guest

src/app/api/booking/confirm/route.ts

The sendBookingConfirmationEmail() function existed in email.ts with a full HTML template, but was never called from the confirm route. After a successful hotel booking, only an admin notification was created — the guest received nothing.

Impact: Hotel guests have no confirmation email with booking ID, property details, dates, or price.

FIXED Added sendBookingConfirmationEmail() call with booking details from result.data and body params. Fire-and-forget pattern (non-blocking).

CRITICAL   Hotel cancellation: NO email sent to guest

src/app/api/booking/cancel/route.ts

No cancellation email function existed for hotels. The cancel route only created an admin notification. Guests were not informed of cancellation status or refund details.

FIXED Created sendHotelCancellationEmail() with refund status (refundable/non-refundable), booking details, and refund amount. Wired into cancel route with DB lookup for property details.

HIGH   Hotel amendment: NO email sent to guest

src/app/api/booking/amend/route.ts

No amendment email existed. Guests were not notified when their booking details were updated.

FIXED Created sendHotelAmendmentEmail() showing updated fields. Wired into amend route.

HIGH   Flight supplier webhook: NO email on ticket issued or refund confirmed

src/app/api/webhooks/flight-supplier/route.ts

When suppliers (Duffel/Mystifly) sent async events for ticket issuance (order.tickets_generated) or refund confirmation (order.cancellation.confirmed), the webhook updated DB status and created admin notifications, but never emailed the passenger.

FIXED Expanded DB select to include passenger_email, passenger_name, segments, total_price, currency. Added sendFlightBookingConfirmationEmail() on ticketed events and sendFlightRefundEmail() on refunded events.

MEDIUM   sendBookingConfirmationEmail returns success: true when email not sent

src/lib/server/email.ts:261

When the Resend API key was not configured, or when Resend returned an error without throwing, the function fell through to return { success: true }. Callers couldn't distinguish between "email actually sent" vs "email silently queued/failed".

FIXED Now returns { success: false, error: 'RESEND_API_KEY not configured' } or { success: false, error: 'Email sending failed' } in those cases.

MEDIUM   Fire-and-forget email pattern with no retry

All email calls use .catch(e => console.error(...)) — errors are logged but never retried. The email_logs table records failures, and the admin dashboard has a manual retry button, but there's no automatic retry pipeline.

Status: Documented. Automatic retry would require a background job/queue system (e.g., pg_cron + Edge Function). Manual retry is available via admin dashboard.

MEDIUM   Dual email logging tables

Hotel emails log to booking_emails table. Flight emails log to email_logs table. Different schemas, different admin queries needed. Creates blind spots in email monitoring.

Status: Documented. Consolidation would require a DB migration (out of scope).

MEDIUM   Cron jobs exist but are not scheduled

supabase/functions/poll-pending-tickets/index.ts
supabase/functions/flight-reconciliation-job/index.ts

Two Supabase Edge Functions exist for ticket polling and flight reconciliation, but the pg_cron scheduling setup is only documented in code comments — never actually configured. Without these cron jobs, awaiting_ticket bookings can expire without auto-refund.

Status: Documented. Requires Supabase SQL editor access to configure pg_cron (infrastructure, not code).

Phase 3 — Monetization & Revenue

Stripe integration, vouchers, pricing, revenue tracking

MEDIUM   Voucher validation lacks rate limiting

src/lib/server/vouchers.ts

The validateVoucherServer() function validates voucher codes against the DB but has no rate limiting. An attacker could brute-force voucher codes by trying thousands of combinations. Server Actions provide CSRF protection but not rate limiting.

MEDIUM   Currency conversion uses static rates

src/lib/currency.ts

Currency conversion uses hardcoded exchange rates. No mechanism to update rates from an external API. Displayed prices may be inaccurate if rates drift significantly.

LOW   No revenue analytics aggregation

The booking_financial_events ledger tracks individual transactions but there's no aggregation layer for total revenue, average booking value, or conversion funnels. The admin dashboard shows raw booking counts but not financial KPIs.

Phase 4 — Admin Dashboard

Admin tools, monitoring, recovery, command palette

MEDIUM   Admin auth relies on hardcoded email check

src/lib/server/admin/

Admin routes check if the authenticated user's email matches an admin list (likely from env vars). No role-based access control (RBAC). If an admin email is compromised, all admin functions are accessible.

LOW   Recovery tools are manual-only

src/lib/server/admin/recovery.ts

Functions like adminForceStatusRecheck(), adminRetryBooking(), and adminCancelAwaitingTicket() exist but require manual admin intervention. The monitoring dashboard detects mismatches (payment without booking after 5 minutes) but doesn't auto-recover.

LOW   No audit log for admin actions

Admin operations (force cancel, force refund, restore booking) update the cancellation_log JSONB array but there's no centralized admin audit log table. Difficult to answer "who did what and when" across different admin actions.

Phase 5 — Search & Discovery

Hotel search, autocomplete, filters, property details

MEDIUM   Search results not cached

Every hotel search calls the LiteAPI Edge Function without caching. Identical searches within seconds hit the API again. React Query provides client-side caching but server-side has none.

LOW   Autocomplete debounce is client-only

src/components/search/DestinationPicker.tsx

Destination autocomplete debounces on the client side but the Edge Function has no rate limiting. Fast typing with poor network could result in multiple concurrent API calls.

LOW   No search analytics

Search queries are not logged. No way to understand what users search for, which searches return empty results, or popular destinations.

Phase 6 — UX & Frontend

Component architecture, accessibility, responsive design, performance

MEDIUM   No loading skeletons on search results

When search results are loading, the UI shows a basic spinner or empty state. No skeleton placeholders that match the card layout, causing layout shift when results appear.

MEDIUM   Checkout form has no field-level validation feedback

The checkout form validates on submit but doesn't show inline validation errors as the user types. Guest name, email, and payment fields could benefit from real-time validation using the existing Zod schemas.

LOW   Missing ARIA labels on interactive elements

Several interactive elements (TiltCard, HorizontalScroll arrows, tab lists) lack proper ARIA labels and roles. Screen readers may struggle with the landing page carousel and filter components.

LOW   Large Framer Motion bundle

Framer Motion is imported in many components. The full library adds ~30KB gzipped. Consider using framer-motion/m (tree-shakeable) or motion (lightweight subset) for simple animations.

The LiteAPI key is stored in Supabase Edge Function environment variables, not exposed to the client. This is correctly implemented. Server Actions and API routes call Edge Functions which add the API key server-side.

MEDIUM   Ownership verification on all booking mutations

cancelBooking, amendBooking, and getBookingDetails all call verifyBookingOwnership() before proceeding. This ensures users can only modify their own bookings.

Phase 9 — Growth & Scalability

Performance bottlenecks, scaling readiness, technical debt

MEDIUM   In-memory concurrency guard doesn't survive server restarts

The inflight Set used for hotel booking double-click prevention is in-memory. If the Next.js server restarts mid-request, the guard is lost. In a multi-instance deployment (e.g., Vercel), each instance has its own Set — so parallel requests to different instances could bypass the guard.

Mitigation: For production multi-instance deployments, consider Redis-based locking or a DB unique constraint on prebookId.

MEDIUM   No database connection pooling configuration

Each API route / Server Action creates a new Supabase client. While Supabase handles connection pooling on their end, the application doesn't configure pool sizes or connection timeouts for high-traffic scenarios.

LOW   No image optimization for property photos

Property and deal images use raw URLs from LiteAPI / static data. No next/image optimization (lazy loading, WebP conversion, responsive srcsets) is applied in cards.

LOW   Bundle size could be reduced

Full imports of framer-motion, lucide-react, and @supabase/supabase-js in client components. Tree-shaking helps but explicit selective imports would reduce initial bundle.
